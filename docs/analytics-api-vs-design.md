# 로그/분석 API vs 디자인(8장 이미지) 비교

이전에 공유해 주신 Figma/이미지 기획과 현재 백엔드 API 응답 구조를 비교한 문서입니다.

---

## 1. 히트맵 (작성 학년 × 열람 학년)

| 구분 | 디자인에서 보통 필요한 것 | 현재 API |
|------|---------------------------|----------|
| **엔드포인트** | - | `GET /api/logs/analytics/heatmap/` |
| **구조** | 4×4 그리드, 행=작성자 학년, 열=열람자 학년, 셀=상호작용 점수 | ✅ 동일 |
| **응답 예시** | - | 아래 참고 |

**현재 API 응답:**
```json
{
  "rows": [
    {
      "author_grade": 1,
      "author_grade_label": "1학년",
      "cols": [
        { "viewer_grade": 1, "viewer_grade_label": "1학년", "score": 42 },
        { "viewer_grade": 2, "viewer_grade_label": "2학년", "score": 31 },
        { "viewer_grade": 3, "viewer_grade_label": "3학년", "score": 28 },
        { "viewer_grade": 4, "viewer_grade_label": "4학년", "score": 19 }
      ]
    },
    { "author_grade": 2, "author_grade_label": "2학년", "cols": [...] },
    { "author_grade": 3, "author_grade_label": "3학년", "cols": [...] },
    { "author_grade": 4, "author_grade_label": "4학년", "cols": [...] }
  ]
}
```

- **매칭:** 행=작성 학년, 열=열람 학년, 셀=가중 합산 점수(조회 1, 좋아요 2, 댓글 3) → **디자인과 구조 일치.**

---

## 2. 학년별 인기 글 (어떤 학년이 어떤 글을 많이 봤는지)

| 구분 | 디자인에서 보통 필요한 것 | 현재 API |
|------|---------------------------|----------|
| **엔드포인트** | - | `GET /api/logs/analytics/popular-by-grade/?top_n=5` |
| **학년별 Top N** | 학년(1~4)별로 “가장 많이 열람/반응한 글” 목록 | ✅ 제공 |
| **글 식별** | 게시판(커뮤니티/네트워크) + 글 ID 또는 제목/링크 | ⚠️ `post_id`만 있음, **section 없음** |

**현재 API 응답:**
```json
{
  "by_grade": [
    {
      "viewer_grade": 1,
      "viewer_grade_label": "1학년",
      "posts": [
        { "post_id": 3, "score": 12 },
        { "post_id": 7, "score": 8 }
      ]
    },
    ...
  ]
}
```

- **갭:** `post_id`만으로는 커뮤니티 3번 글인지 네트워크 3번 글인지 구분 불가.  
  → **권장:** 각 항목에 `section` (`"community"` | `"network"`) 추가.

---

## 3. 인기 검색어 Top 10

| 구분 | 디자인에서 보통 필요한 것 | 현재 API |
|------|---------------------------|----------|
| **엔드포인트** | - | `GET /api/logs/analytics/search-ranking/?section=community&top_n=10` |
| **순위/키워드/검색 횟수** | 순위, 키워드, 검색 횟수 | ✅ 제공 |
| **주요 검색 학년** | 해당 키워드를 가장 많이 검색한 학년 | ✅ `main_grade`, `main_grade_label` 제공 |

**현재 API 응답:**
```json
{
  "ranking": [
    {
      "rank": 1,
      "keyword": "취업",
      "count": 42,
      "main_grade": 3,
      "main_grade_label": "3학년"
    },
    ...
  ]
}
```

- **매칭:** Top N, 키워드, 횟수, 주요 학년 → **디자인과 일치.**

---

## 4. XP(기여도) 랭킹

| 구분 | 디자인에서 보통 필요한 것 | 현재 API |
|------|---------------------------|----------|
| **엔드포인트** | - | `GET /api/logs/analytics/xp-ranking/?top_n=20` |
| **XP 공식** | (글×10)+(댓글×5)+(채택답×20)+(졸업후기×50) | ⚠️ 현재는 (글×10)+(댓글×5)만 반영 |
| **표시용 정보** | 순위, 닉네임/이름(선택), XP, 글/댓글 수 | ⚠️ `author_id`만 있음, **닉네임 없음** |

**현재 API 응답:**
```json
{
  "ranking": [
    {
      "rank": 1,
      "author_id": 5,
      "post_count": 10,
      "comment_count": 30,
      "xp": 250
    },
    ...
  ]
}
```

- **갭 1:** 채택 답변(C), 졸업/대학원 후기(D)는 미구현 → DB에 해당 구분 필드 추가 후 반영 가능.  
- **갭 2:** UI에 “누가 몇 위인지” 보여주려면 `author_id`만으로는 부족.  
  → **권장:** 관리자용이면 `author_id`로 충분할 수 있고, 노출용이면 닉네임(또는 익명 처리) 등 추가 API/필드 필요.

---

## 5. 그 외 (디자인에 있었다면)

| 항목 | 현재 API | 비고 |
|------|----------|------|
| **페이지별 방문 수** | ❌ 전용 API 없음 | `EventLog`에 `page_view` + `section`/`page` 있으므로 집계 API 추가 가능 |
| **학년별 활동량(글/댓글 수)** | ❌ 전용 API 없음 | 로그 또는 User/Post JOIN으로 구현 가능 |
| **기간 필터(오늘/7일/30일)** | ❌ 없음 | 모든 API에 `?from=&to=` 또는 `?period=7d` 추가 가능 |

---

## 요약

| 화면/기능 | API 일치 여부 | 보완 제안 |
|-----------|----------------|-----------|
| 히트맵 | ✅ 일치 | - |
| 인기 검색어 | ✅ 일치 | - |
| 학년별 인기 글 | ⚠️ 거의 일치 | 응답에 `section` 추가 |
| XP 랭킹 | ⚠️ 부분 일치 | XP 공식 확장(C,D), 노출용이면 닉네임 등 추가 |
| 페이지별 방문/학년별 활동 | ❌ API 없음 | 필요 시 새 엔드포인트 추가 |
| 기간 필터 | ❌ 없음 | 쿼리 파라미터로 추가 |

8장 이미지가 있으면 화면 단위로 “이 블록은 이 API 필드로 매핑한다”까지 더 구체적으로 적어줄 수 있습니다. 이미지 다시 공유해 주시면 그걸 기준으로 보완안 정리해 드리겠습니다.
